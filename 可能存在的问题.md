完整审查报告
我已完成对整个项目的逐文件审查。以下是发现的问题。

一、错误与不符合开发规范的问题
1. [严重] app.py 第8行提前导入导致 OWNER_SALT 读取不到 .env 配置

app.py:8 在 load_dotenv() (第12行) 之前导入了 modules.auth.service。这把 auth/service.py 中所有模块级变量求值提前到了 .env 加载之前：


# auth/service.py:30-31 — 这两行在 app.py 第8行就求值了，比第12行的 load_dotenv() 早
OWNER_SALT = os.getenv("AUTH_OWNER_SALT", os.getenv("SECRET_KEY", "dev-secret-key"))
SESSION_HOURS = max(1, int(os.getenv("AUTH_SESSION_HOURS", "8")))
这意味着用户即使在 .env 中设置了 AUTH_OWNER_SALT，它也会被忽略，OWNER_SALT 永远是 "dev-secret-key"。build_owner_identity() 和 build_external_identity() 产生的哈希身份将不会使用用户设定的盐值。

2. [严重] AI 与 Feature3 模块之间构成双向依赖

ai/routes.py:11：AI 路由模块导入 feature3.service 的 get_feature3_runtime_summary
feature3/service.py:9：Feature3 服务模块导入 ai.service 的 build_answer
两个本来应独立的模块互相导入，增加耦合。/api/demo_status 是在 AI 路由中注册却需要展示 Feature3 状态，这是导致这种交叉依赖的根本原因。

3. [中等] 文本规范化/分词函数在3个文件中重复实现

函数	所在文件
normalize_text / _normalize_text	ai/service.py:84-88, ai/kb.py:148-152
char_ngrams / _char_ngrams	ai/service.py:91-94, ai/kb.py:155-158, ai/web_fetcher.py:363-367
2-gram 评分逻辑	ai/service.py:97-112 (score_question), ai/kb.py:161-176 (_lexical_score)
这三组函数实现完全相同，只是下划线前缀不同（代表"私有"），违反 DRY 原则。应该统一提取到 shared 工具中。

4. [中等] generate_view_code() 与 create_question() 重复生成逻辑

feature3/service.py:126-127 的 generate_view_code() 和 shared/db.py:143 的 create_question() 内部都在做 secrets.token_hex(4).upper()。service 层生成 code 后传给 db 层，而 db 层在 code 为 None 时又自己生成一遍——两层重复了同一个默认值逻辑。

5. [中等] PERM_ADMIN 已定义但从未被任何路由/守卫使用

auth/service.py:20 定义了 PERM_ADMIN = 4，auth_identities.json.example 中也有示例管理员条目。但全局代码中没有任何一处 require_permission(PERM_ADMIN) 调用。权限位已定义但完全没有落地使用。api_feature3_status 虽返回 is_admin，但没有任何接口实际检查该权限。

6. [中等] auth/service.py 中 SESSION_HOURS 变量未被使用

auth/service.py:31 定义了 SESSION_HOURS，但 login_user() 只设置了 session.permanent = True，真正的 session 有效期由 app.py:18 的 app.config["PERMANENT_SESSION_LIFETIME"] 控制。auth/service.py 中的 SESSION_HOURS 是死代码。

7. [低] chat.html 中包含大量未使用的 CSS 规则

chat.html:551-613 定义了 .inspector、.timeline、.source-list、.source-card、.logic-list 等样式规则，但 HTML 中完全没有任何对应元素。这是从旧版界面（带有过程可视化面板的版本）残留的死 CSS，合计约60行。

8. [低] 每个 HTML 页面各自重复实现 esc() 和 fmtTs() 函数

esc() / escapeHtml() 出现在 senior.html:217-219、freshman.html:368-371、chat.html:882-889
fmtTs() 出现在 senior.html:226-233、freshman.html:292-300
可以考虑抽取一个共享 JS 文件，但鉴于当前项目刻意不使用前端构建工具，这个重复可以接受。

二、冗余设计与不合理的耦合
1. official_url_allowlist.json 是死配置文件

official_url_allowlist.json 手工整理了12个 BIT 官方子域名和5个受限站点，还标注了 verified、crawl_enabled、category 等字段。但全局没有任何代码 import json 读取这个文件。web_fetcher.py 中的站点信任逻辑完全使用的是硬编码列表（_LOW_TRUST_HOST_SUFFIXES、_SEARCH_PROVIDER_HOST_SUFFIXES 等），与这个 JSON 文件毫无关联。

2. /api/demo_status 造成模块间不必要的耦合

ai/routes.py:63-78 — api_demo_status 注册在 AI 路由中，但它聚合了 AI 状态 + Feature3 状态 + 路由清单。这迫使 AI 模块必须导入 Feature3 模块。按照开发文档建议的"保持模块职责单一"原则，这个聚合端点应该放在 app.py 层或单独的 diagnostics 模块中。

3. pages/routes.py 中5个路由可以使用映射表简化

pages/routes.py:6-28 — /、/chat、/place、/freshman 四个路由是完全相同的模式，仅文件名不同。当前写法每个路由都要手写一个函数，可简化为：


PAGES = {
    "/": "index.html",
    "/chat": "chat.html",
    "/place": "place.html",
    "/freshman": "freshman.html",
}
for path, filename in PAGES.items():
    app.add_url_rule(path, view_func=...)
当前写法虽然代码更少但模板重复。

4. kb.py 中 query_index 的延迟索引 bootstrap 引入隐藏副作用

kb.py:388-393 — 当 load_index() 返回 None（索引文件缺失）时，query_index 会自动调用 build_index() 重建索引。这意味着一个"查询"操作可能触发"全量重建"的写操作，对调用者完全透明。这违反了查询-命令分离原则，且可能在首次查询时造成长延迟。

5. kb.py 中 _get_model() 使用模块级全局变量可能导致线程问题

kb.py:313-322 — _MODEL 是模块级单例，_get_model() 在 if _MODEL is None 条件下的赋值不是线程安全的。虽然 Flask 开发服务器单线程运行时无影响，但在 Gunicorn 多 worker 或多线程场景下可能导致竞态条件。

6. web_fetcher.py 中 _ROBOTS_CACHE 使用不限制大小的全局字典

web_fetcher.py:39 — _ROBOTS_CACHE 是一个无界增长的全局字典，以域名为键。在长期运行的进程中，如果用户查询涉及多个不同域名，缓存会无限增长。虽然有 TTL 过期逻辑，但过期条目不会被清理，只是不再返回。
