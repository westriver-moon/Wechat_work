# 功能三开发完整流程文档（无登录版）

## 1. 项目目标与边界

### 1.1 目标
实现“新生提问 -> 学长回复 -> 用户主动拉取查看”的人工工单系统，满足未认证公众号场景。

### 1.2 边界
- 不做登录/OAuth/JWT 设计。
- 采用无登录 MVP：
  - 新生端用 `client_id`（浏览器本地标识）管理“我的问题”。
  - 学长端用 `X-Senior-Key` 作为临时密钥鉴权。
- 公众号不依赖主动客服推送，主路径是 H5 页面主动拉取。

---

## 2. 需求拆解

### 2.1 基础需求
1. 提交问题
2. 问题入库与状态管理（pending/answered/closed）
3. 学长查看待办并回复
4. 新生查看“我的问题”并关闭工单
5. 公众号消息与工单联动（编号+查询码）

### 2.2 增强需求（本轮新增）
1. 学长端增加“已回答列表”
2. 学长端增加搜索（编号/提问/回答/查询码）
3. 新生端自动轮询
4. 新生端新回答高亮提醒

---

## 3. 架构设计

### 3.1 后端
- 框架：Flask
- 存储：SQLite
- 模块：
  - `app.py`：路由与业务编排
  - `db.py`：数据访问层
  - `wechat_utils.py`：可选主动推送工具（默认关闭）

### 3.2 前端
- `freshman.html`：新生提问、查看、查询、关闭
- `senior.html`：学长待办处理、已回答查看与搜索

### 3.3 数据模型
- `users`
- `questions`
- `answers`
- `tasks`

---

## 4. 关键实现过程

### 4.1 数据层（db.py）
1. 建表初始化 `init_db()`
2. 用户创建/查询 `get_or_create_user()`
3. 问题创建 `create_question()`
4. 问题列表
   - `list_questions_by_client()`
   - `list_pending_questions_with_query()`
   - `list_answered_questions()`
5. 回答与状态更新 `save_answer()`
6. 关闭问题 `close_question()`
7. 任务队列 `enqueue_task()/claim_next_task()/mark_task_done()/mark_task_retry()`

### 4.2 API 层（app.py）
- 新生侧：
  - `POST /api/questions`
  - `GET /api/questions/mine`
  - `GET /api/questions/track`
  - `PUT /api/questions/<id>/close`
- 学长侧：
  - `GET /api/tasks/pending?q=...`
  - `GET /api/tasks/answered?q=...`
  - `POST /api/answers`
- 运行状态：
  - `GET /api/feature3_status`

### 4.3 微信消息（/wechat）
- 收到文本后：
  1. 如果是查询格式（查 编号 查询码），返回状态/答案摘要
  2. 否则创建工单并返回“编号+查询码+H5提示”
- 避免同步长耗时，保证微信侧稳定

### 4.4 学长端增强（senior.html）
- 页面分为两列：
  - 待处理问题
  - 已回答问题
- 增加关键字搜索输入，统一作用于待办与已答列表
- 回复提交后自动刷新两个列表

### 4.5 新生端增强（freshman.html）
- 自动轮询（默认 12 秒）
- 本地维护已见回答时间戳（localStorage）
- 若发现新回答：
  - 顶部出现提醒
  - 对对应问题卡片高亮

## 5. 运行配置

在 `.env` 中建议配置：

```env
FEATURE3_TICKET_MODE=1
FEATURE3_AUTO_AI_ANSWER=0
FEATURE3_ENABLE_WORKER=1
FEATURE3_WORKER_POLL_SECONDS=1.5
FEATURE3_SENIOR_KEY=dev-senior-key
FEATURE3_HELP_URL=/freshman
FEATURE3_ENABLE_WECHAT_PUSH=0
DB_PATH=./feature3.db
```

说明：
- `FEATURE3_AUTO_AI_ANSWER=0` 表示纯人工回复。
- 若设为 `1`，worker 可自动生成 AI 回复。

## 8. 已知限制与后续演进

### 8.1 当前限制
- 无登录模型，仅适合 MVP/小规模场景
- 学长密钥方案不适合长期生产
- SQLite 在高并发写入下有上限

### 8.2 建议演进
1. 接入正式鉴权（OAuth/JWT/RBAC）
2. 数据库迁移至 MySQL/PostgreSQL
3. 增加操作审计与管理后台
4. 增加告警与监控（服务可用性、处理时延）
