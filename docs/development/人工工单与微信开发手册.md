# 人工工单与微信开发手册

## 1. 功能定位

人工工单模块用于承接“AI 不能完全覆盖”或“需要人工处理”的问题链路。它既支持网页登录态下的新生提交与学长回复，也支持微信公众号文本消息进入同一套工单系统。

当前能力包括：

- 新生网页提交问题
- 查看当前账号下的问题列表
- 使用问题编号 + 查询码追踪问题
- 学长查看待处理问题并提交答复
- 答复后问题转为已回答
- 问题可由发起者关闭
- 微信文本消息进入工单流程
- worker 可选自动生成 AI 答复

## 2. 代码结构

### 路由层

`app/backend/modules/feature3/routes.py`

负责：

- 请求参数校验
- 登录态和权限判断
- 调用 service / db 层
- 封装 JSON 或 XML 响应

### 业务层

`app/backend/modules/feature3/service.py`

负责：

- worker 生命周期管理
- 任务处理
- 查询码生成
- 微信签名校验
- 微信文本转工单逻辑
- feature3 状态汇总

### 微信辅助

`app/backend/modules/feature3/wechat_utils.py`

负责：

- access_token 缓存
- 主动客服消息推送

### 数据层

`app/backend/modules/shared/db.py`

负责：

- 工单相关表的持久化
- 用户、问题、答案、任务、身份白名单等查询与更新

## 3. 核心数据模型

当前工单系统主要依赖以下表：

### `users`

记录系统内的用户实体，包括：

- 登录态用户映射后的 `owner_identity`
- 学长或管理员身份
- 微信来源用户

### `questions`

记录问题主体：

- 提问者
- 标题
- 内容
- 状态
- 查询码
- 来源渠道

### `answers`

记录最终答复：

- 对应问题
- 答复人
- 答复类型（人工 / AI）
- 答复内容

### `tasks`

记录异步任务：

- 任务类型
- payload
- 尝试次数
- 失败原因
- 当前状态

### `auth_identities`

不是工单专属表，但会影响工单后台权限。

## 4. 状态流转

### 问题状态

当前主要状态：

- `pending`
- `answered`
- `closed`

### 典型流程

#### 网页提问流程

1. 登录用户在 `/freshman` 提交问题
2. `POST /api/questions` 创建问题
3. 问题状态初始化为 `pending`
4. 若启用自动 AI 答复，则入队任务
5. 学长在 `/senior` 查看待处理问题
6. `POST /api/answers` 提交答复
7. 问题状态转为 `answered`
8. 提问者关闭问题后转为 `closed`

#### 微信文本流程

1. 微信发送普通文本到 `/wechat`
2. 校验签名成功后解析消息
3. 若符合“查 编号 查询码”格式，直接回查状态
4. 否则创建新工单并返回问题编号和查询码
5. 后续通过网页或微信查询继续跟踪

## 5. 路由清单

### 新生侧

- `POST /api/questions`
- `GET /api/questions/mine`
- `GET /api/questions/track`
- `PUT /api/questions/<id>/close`

### 学长侧

- `GET /api/tasks/pending`
- `GET /api/tasks/answered`
- `POST /api/answers`

### 运行态

- `GET /api/feature3_status`

### 微信回调

- `GET /wechat`
- `POST /wechat`

## 6. 登录与权限关系

### 新生侧

当前网页提问依赖登录态，并通过 `owner_identity` 绑定“我的问题”。

### 学长侧

学长处理台依赖 `PERM_ANSWER` 权限。没有该权限的登录用户即使能登录，也不能访问 `/senior` 或答复相关接口。

### 微信侧

微信用户不依赖网页登录态，系统会把其映射为外部身份并创建对应用户实体。

## 7. worker 机制

worker 在应用启动时由 `start_feature3_worker(app)` 拉起。

主要职责：

1. 轮询 `tasks` 表
2. 领取待处理任务
3. 调用 `_process_task`
4. 成功后标记完成
5. 失败后记录错误并按次数决定重试或失败

当前主要任务类型是：

- `ai_answer`

如果后续需要扩展提醒、审计、通知等能力，建议继续沿用 `tasks` 表而不是临时增设线程逻辑。

## 9. 常见改造场景

### 扩展问题字段

如果要新增标签、优先级、附件等信息：

1. 先调整 `shared/db.py`
2. 再调整 `routes.py` 的请求和响应结构
3. 最后调整前端表单和列表展示

### 扩展学长处理能力

如果要新增分派、认领、批量处理：

1. 先明确状态机
2. 再增加相应字段和接口
3. 不建议直接在现有 `answered/closed` 上做隐式复用

### 扩展微信能力

如果要新增模板消息或更复杂的客服交互：

1. 先确认公众号能力和认证状态
2. 再扩展 `wechat_utils.py`
3. 保持 `/wechat` 的请求处理足够轻量，避免同步阻塞

## 10. 自动化与手工验证

### 自动化冒烟测试

```powershell
Set-Location .\app\backend
D:\envs\wechat_work\python.exe .\tests\feature3_smoke_test.py
```

### 最低手工验证

1. 登录后访问 `/freshman`
2. 提交一条问题
3. 登录有答复权限的账号访问 `/senior`
4. 提交答复
5. 回到 `/freshman` 确认状态和答复展示
6. 验证关闭问题后不能再次答复

### 微信验证

如果已配置公众号：

1. 校验 `/wechat` GET
2. 发送普通问题
3. 发送 `查 编号 查询码`
4. 验证回查结果与网页状态一致

## 11. 当前限制

- 学长权限仍依赖本地白名单配置
- SQLite 适合低并发和单实例场景
- 微信主动推送依赖外部配置，不是默认主路径
- 工单与登录系统共用同一应用实例，改动时要注意边界，不要互相污染

## 12. 后续演进建议
