# 后端开发手册

## 1. 启动入口与装配方式

后端唯一启动入口是 `app/backend/app.py`。

启动时完成以下动作：

1. 从 `app/backend/.env` 加载环境变量。
2. 创建 Flask 应用，并把 `app/web` 作为静态目录。
3. 初始化 SQLite 数据库。
4. 从 `auth_identities.json` 同步特权身份。
5. 启动 feature3 worker。
6. 注册认证、页面、AI、工单四组路由。

这意味着：

- 新模块接入必须最终由 `app.py` 注册。
- 不能假设某个子模块可以独立成为主入口。
- `app.py` 负责装配，不应堆积业务细节。

## 3. 认证与权限模型

### 登录模型

当前后端默认要求登录后访问大部分页面和 API。

未登录时：

- 页面请求重定向到 `/login`
- API 请求返回 `401` JSON

### 权限位

当前权限位定义在认证模块中：

- `PERM_USE_APP`
- `PERM_ANSWER`
- `PERM_ADMIN`

使用方式：

- 基础登录用户拥有 `PERM_USE_APP`
- 学长答复能力依赖 `PERM_ANSWER`
- 管理类动作依赖 `PERM_ADMIN`

### 特权身份

特权身份来源于 `app/backend/auth_identities.json`，启动时同步到数据库。用途主要是：

- 为特定学号配置学长答复权限
- 为管理员保留更高权限
- 在不污染普通登录用户数据的前提下附加角色能力

## 6. 新增后端功能时的建议

### 新增 API

建议步骤：

1. 判断功能属于哪个模块。
2. 优先在对应 `service.py` 实现业务逻辑。
3. 在 `routes.py` 中只保留参数校验、权限判断和响应封装。
4. 如果涉及持久化，优先在 `shared/db.py` 增补最小必要接口。
5. 最后补一条自动化或手工验证路径。

### 新增页面

建议步骤：

1. 先在 `app/web/` 增加页面文件。
2. 再在 `modules/pages/routes.py` 增加入口。
3. 若页面需要受保护或限制权限，直接复用认证模块已有能力。

### 新增 worker 任务

建议步骤：

1. 在 `tasks` 表中定义新的 `type`。
2. 在 `modules/feature3/service.py` 的 `_process_task` 中处理。
3. 确保失败时可重试且不会造成不可逆副作用。

## 7. 排障入口

出现后端问题时，优先按以下顺序排查：

1. 服务能否成功导入 `from app import app`
2. `.env` 是否已正确加载
3. `feature3.db` 是否可写
4. `/api/demo_status` 是否返回合理状态
5. `/api/chat` 是否在登录态下返回 200
6. `tests/` 下的两条冒烟脚本是否通过

## 8. 不建议的做法

- 不要在 backend 根目录重新引入平铺模块文件。
- 不要在 `app.py` 中直接堆业务逻辑。
- 不要在页面 JS 中硬编码后端内部实现细节。
- 不要把调试临时脚本长期留在 backend 根目录。
- 不要让 AI 逻辑直接跨层操作工单状态；应通过 service 层明确调用。
