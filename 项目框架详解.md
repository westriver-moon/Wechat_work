# 项目框架详解

本文档面向当前项目的协作开发者，目标不是简单列目录，而是解释项目为何这样组织、模块之间如何协作、请求如何流动、哪些地方是扩展点，哪些地方是边界。

## 1. 项目定位

项目服务于“新生助手”这一综合场景，不是单一功能站点。当前系统同时承担四类职责：

1. 登录与访问控制
2. AI 问答与知识库检索
3. 人工工单与微信回调
4. Web 页面与基础导航

换句话说，这个仓库不是“AI demo”，也不是“工单 demo”，而是一个把登录、问答、地图、工单和公众号接入放在同一应用实例中的统一系统。

## 2. 产品视角下的角色划分

### 普通登录用户

主要能力：

- 登录后访问首页
- 使用 AI 问答
- 使用地点查询
- 在 `/freshman` 中提交和查看自己的问题

### 学长 / 答复人

在普通登录能力之上，额外具备：

- 访问 `/senior`
- 查看待处理工单
- 提交答复
- 查看已答复问题列表

### 管理员

当前代码中管理员能力主要体现在更高权限位，但并未单独做出完整后台。

### 微信用户

不一定具备网页登录态。微信文本消息会通过 `/wechat` 进入系统，并映射为外部身份用户。

## 3. 为什么采用当前目录结构

当前目录结构的核心目标是：

- 后端入口唯一
- 功能边界明确
- 测试脚本统一收口
- 历史文档与当前文档隔离
- runtime 文件和源码结构分离理解

因此 backend 采用如下形式：

```text
app/backend/
├── app.py
├── modules/
├── tests/
├── .env.example
├── requirements.txt
└── 其他配置文件
```

这个布局有三个直接好处：

### 1. `app.py` 只承担装配责任

它负责：

- 加载 `.env`
- 创建 Flask 应用
- 初始化数据库
- 启动 worker
- 注册路由

它不承担业务实现，从而避免再次回到“大单文件堆所有逻辑”的状态。

### 2. 真实实现全部进入 `modules/`

后续维护者找代码时，不需要猜“真实逻辑在根目录还是在某个迁移临时文件里”，只看 `modules/` 即可。

### 3. 自动化测试统一进入 `tests/`

测试脚本不再散落在 backend 根目录，从结构上就能区分“运行入口”和“验证入口”。

## 4. 应用启动过程

启动入口：`app/backend/app.py`

启动顺序如下：

1. 定位并加载 `app/backend/.env`
2. 创建 Flask 对象，静态目录指向 `app/web`
3. 设置 Session 相关配置
4. 调用 `init_db()` 建表
5. 尝试同步 `auth_identities.json`
6. 调用 `start_feature3_worker(app)` 拉起后台 worker
7. 注册认证路由
8. 注册页面路由
9. 注册 AI 路由
10. 注册工单 / 微信路由

这个顺序反映了项目的真实依赖关系：

- 页面和 API 都依赖认证守卫
- 工单 worker 依赖数据库
- AI 和工单都共享同一个应用实例

## 5. 路由分层

### 认证路由

来自 `modules/auth/routes.py`：

- `/login`
- `/logout`
- `/api/auth/login`
- `/api/auth/logout`
- `/api/auth/me`

此外还注册 `before_request`，它决定哪些路径公开、哪些路径必须登录。

### 页面路由

来自 `modules/pages/routes.py`：

- `/`
- `/chat`
- `/place`
- `/freshman`
- `/senior`

页面本身是静态文件，但访问权由后端路由控制。

### AI 路由

来自 `modules/ai/routes.py`：

- `POST /api/chat`
- `POST /api/rebuild_index`
- `GET /api/kb_query`
- `GET /api/demo_status`

### 工单与微信路由

来自 `modules/feature3/routes.py`：

- `POST /api/questions`
- `GET /api/questions/mine`
- `GET /api/questions/track`
- `PUT /api/questions/<id>/close`
- `GET /api/tasks/pending`
- `GET /api/tasks/answered`
- `POST /api/answers`
- `GET /api/feature3_status`
- `GET/POST /wechat`

## 6. 认证模型

### Session 是所有 Web 功能的基础

当前项目不是前后端分离的 token 模式，而是：

- 登录接口创建 Session
- 页面访问依赖 Session
- API 访问也依赖 Session

### 用户对象来源

登录成功后，系统会构造 Session 用户对象，包含：

- 学号
- 姓名
- 专业等信息
- `permissions`
- `permission_labels`
- `owner_identity`
- `is_privileged`

其中 `owner_identity` 很关键，它把“登录用户”和“工单归属”绑定起来，是 `/api/questions/mine` 能正确工作的前提。

### 权限附加方式

本地并不为所有登录用户做完整权限落库，而是只维护少量特权身份。这样做的目的：

- 普通用户只需要可登录和使用基本功能
- 学长 / 管理员通过白名单附加权限位
- 避免把权限系统做得过重

## 7. AI 模块的真实工作方式

当前 AI 不是“固定问答库”模式，而是三层组合：

1. 高频问题缓存
2. 本地知识库检索
3. 公开网页搜索与抓取
4. 可选 LLM 组织最终回答

### AI 请求流

当用户调用 `/api/chat`：

1. 路由层校验问题非空
2. 进入 `build_chat_response`
3. 如果网页抓取开启，先做全网搜索并抓取候选网页
4. 如果有可用网页结果，则基于网页内容生成回答
5. 如果网页结果不足，则回退到本地知识库
6. 如果配置了 LLM，就让模型基于检索内容生成自然语言回答
7. 如果没有 LLM，就直接输出检索结果或兜底文案

### 为什么要保留知识库回退

因为网页抓取不是总能成功：

- 搜索引擎可能超时
- 目标网站可能不可抓取
- 内容可能不够稳定
- 网络可能受限

因此知识库回退不是备选功能，而是稳定性的一部分。

### 为什么 `app/data/` 不移动

`app/data/*.md` 虽然也是 Markdown，但它们是运行时知识数据，不是项目文档。AI 的检索质量直接依赖这些文件，因此它们应继续留在 `app/data/`，而不是被归到 `docs/`。

## 8. 工单系统的真实工作方式

工单系统既不是单纯的 CRUD，也不是单纯的聊天记录，而是一套状态流转系统。

### 核心状态

- `pending`
- `answered`
- `closed`

### 网页工单流

1. 登录用户在 `/freshman` 提交问题
2. 系统创建 `questions` 记录
3. 问题进入 `pending`
4. 有权限的学长在 `/senior` 查看待办
5. 学长提交答复后，写入 `answers` 并把问题转为 `answered`
6. 发起者确认结束后，问题转为 `closed`

### 微信文本流

1. 微信文本进入 `/wechat`
2. 系统校验签名
3. 如果是 `查 编号 查询码` 形式，则回查问题状态
4. 否则创建新工单
5. 返回“问题编号 + 查询码 + 页面查看提示”

### worker 的位置

worker 由 feature3 模块启动，但它不是独立服务，而是同一应用进程中的后台线程。当前主要用途是处理 AI 自动答复任务。

这意味着：

- 功能足够简单时部署方便
- 但如果后续任务类型增多，可能需要进一步独立队列系统

## 9. 数据层设计

共享数据访问层位于 `modules/shared/db.py`。

当前项目中，工单和认证附加能力共用同一个 SQLite 文件，这是一种“轻量单实例”设计。

### 主要表

- `users`
- `questions`
- `answers`
- `tasks`
- `auth_identities`

### 这种设计的优点

- 本地开发成本低
- 部署简单
- 不需要额外数据库服务

### 这种设计的限制

- 高并发写入能力有限
- 多实例扩展难
- worker 和业务请求共享同一文件锁体系

因此，它适合当前阶段，但不是长期高并发方案。

## 10. 页面系统与后端的关系

当前前端页面全部由 Flask 静态托管，而不是独立前端工程。这种设计带来的特征是：

- 页面部署简单
- 路由和权限能在一处控制
- 不需要单独打包前端构建系统

同时也带来约束：

- 页面复杂度不能无限膨胀
- 页面状态管理主要靠原生 JS
- API 合同变化会直接影响页面脚本

### 页面职责分工

- `login.html`：登录与错误提示
- `index.html`：统一入口
- `chat.html`：问答输入与结果展示
- `place.html`：地点查询和地图渲染
- `freshman.html`：问题提交、我的问题、查询码追踪
- `senior.html`：待办、已答复、答复提交

## 11. 配置体系

### `.env` 的地位

`app/backend/.env` 是运行时配置的核心来源。无论本地还是生产，都应以 `.env.example` 为模板。

### 配置分组

- 应用与 Session：`SECRET_KEY`、`AUTH_SESSION_HOURS`
- AI：`DEEPSEEK_*`、`OPENAI_*`、`LLM_*`
- 检索：`KB_*`
- 网页抓取：`WEB_FETCH_*`、`WEB_SEARCH_*`
- 工单：`FEATURE3_*`
- 微信：`WECHAT_TOKEN`、`WECHAT_APPID`、`WECHAT_APPSECRET`

### 需要手动修改但不在 `.env` 中的项

- `app/web/place.html` 中的腾讯地图 Key

## 12. 部署拓扑与运行边界

生产推荐拓扑：

- Nginx 对外
- Gunicorn 托管 Flask
- systemd 管理 Gunicorn
- SQLite 作为本地存储

### 为什么 Nginx 超时要保守设置

AI 请求可能会经历：

1. 搜索引擎请求
2. 候选网页抓取
3. HTML 正文提取
4. LLM 生成

因此代理超时不能按普通页面接口来设，否则浏览器层会先报错，用户看到的只是 HTTP2 协议错误，而不是应用层信息。

## 13. 测试体系

当前自动化测试刻意保持为两条核心冒烟脚本：

- `tests/feature3_smoke_test.py`
- `tests/url_fetch_smoke_test.py`

它们覆盖的不是全部细节，而是当前最关键、最容易被重构破坏的两条链路：

- AI 网页抓取能力
- 工单状态流转能力

除此之外，还需要配合浏览器回环测试：

- 登录页可打开
- AI 页可回答
- 工单页可提交和查看
- 地图页可加载

## 14. 当前最重要的边界条件

### 1. 地图功能不是纯后端能力

地图页是否真正可用，除了代码，还依赖腾讯地图 Key 和来源限制配置。

### 2. BIT 登录不是完全本地可控

登录链路依赖外部认证服务和依赖安装来源，测试时要区分“接口能工作”和“真实账号能登录成功”。

### 3. AI 是否能返回高质量回答，取决于外部条件

包括：

- LLM Key 是否可用
- 搜索引擎是否能访问
- 目标网页是否可抓取
- 本地知识库是否足够

### 4. 工单系统与登录系统存在天然耦合，但不应相互污染

工单页面依赖登录态和权限，工单归属依赖 `owner_identity`，但工单模块本身不应反向改写登录模型。

## 15. 扩展建议

### 如果要新增功能四

建议优先判断：

- 它是页面型功能、AI 型功能，还是工单型功能
- 是否能落入已有模块
- 是否真的需要单独新建模块

### 如果要做更正式的后台

建议方向：

- 把 `admin` 权限真正落成管理界面
- 增加操作审计
- 增加用户和权限管理页面

### 如果要提升生产能力

建议方向：

- SQLite 迁移到 MySQL / PostgreSQL
- worker 从线程迁移到独立任务队列
- 对登录和工单做更细粒度的审计与监控

## 16. 协作开发建议

### 看代码的顺序

建议新同学按这个顺序入手：

1. `README.md`
2. `项目结构总览.md`
3. `app/backend/app.py`
4. `modules/auth/`
5. `modules/pages/`
6. `modules/ai/`
7. `modules/feature3/`
8. `tests/`

### 修改时的原则

- 先确认功能归属，再修改模块
- 不在根目录补临时兼容层
- 修改后立刻做最小回归
- 文档变化和结构变化同步提交

## 17. 总结

当前项目已经从早期的功能叠加状态，进入到“统一入口 + 模块分层 + 测试收口 + 文档分层”的结构阶段。它仍然是一个单体应用，但已经具备继续演进的边界条件：

- 登录单独分层
- AI 单独分层
- 工单单独分层
- 页面与共享基础设施分层
- 历史文档与当前文档分层

对协作开发者来说，最重要的不是记住所有细节，而是先理解：

- 入口在哪里
- 状态存在哪里
- 路由由谁注册
- 权限由谁控制
- AI 和工单分别如何闭环

理解了这五件事，后续维护就会稳定很多。
