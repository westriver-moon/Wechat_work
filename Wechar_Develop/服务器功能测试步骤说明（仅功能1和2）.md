# 服务器功能测试步骤说明（仅功能1和2，含网页演示流程）

适用范围：
- 功能一 地点查询
- 功能二 智能问答

不包含：功能三（人工服务）

---

## 1. 测试前准备

- 已完成部署并可公网访问
- 已配置公众号服务器 URL 与 Token
- 已确认 Nginx 与后端服务正在运行
- 准备 1 台手机（关注公众号）+ 1 台电脑（浏览器 + 可看日志）

建议记录：
- 测试时间
- 测试人
- 环境（域名、版本）

## 2. 基础连通测试

### 2.1 页面可达性

依次访问：
- `https://你的域名/`
- `https://你的域名/place`
- `https://你的域名/chat`
- `https://你的域名/api/demo_status`

预期结果：
- 页面均可打开，且移动端样式无明显错位。
- `/api/demo_status` 返回 JSON，包含 `provider`、`llm_enabled`、`model`、`kb_index_ready`。

补充：统一入口页 `https://你的域名/` 需可直接看到两个功能（地点查询 + 智能问答）的访问区域。

### 2.2 微信接口可达性

在公众号后台点击“提交/校验服务器配置”。

预期结果：
- 校验通过，无签名错误。

## 3. 功能一测试：地点查询

### 3.1 正常路径

1. 打开 `https://你的域名/place`
2. 输入 `北京理工大学`
3. 点击“查询地点”

预期结果：
- 地图中心移动到目标位置
- 出现标记点
- 名称、地址、坐标更新
- 静态图加载成功

### 3.2 异常路径

1. 输入空字符串并提交
预期：提示“请先输入地点关键词”

2. 输入无意义关键词（如 `asdasdxxxx`）
预期：提示“未找到匹配地点”

3. 若提示“地图 Key 当日调用量已达上限”
说明：腾讯地图配额已用尽，属于配额问题，不是代码崩溃。

## 4. 功能二测试：智能问答

### 4.0 索引重建验证（汇报前必测）

1. 调用：

```bash
curl -X POST 'https://你的域名/api/rebuild_index'
```

2. 再调用：

```bash
curl 'https://你的域名/api/demo_status'
```

预期结果：
- `/api/rebuild_index` 返回 `ok=true`。
- `/api/demo_status` 显示 `kb_index_ready=true`，且 `kb_doc_count` 大于 0。
- 当 `.env` 中 `KB_FORCE_LEXICAL=0` 时，`kb_backend` 应为 `faiss`。

Windows 命令替代：

```powershell
Invoke-RestMethod -Method Post -Uri 'https://你的域名/api/rebuild_index'
Invoke-RestMethod -Method Get -Uri 'https://你的域名/api/demo_status'
```

### 4.1 接口级测试

使用命令测试：

```bash
curl -X POST 'https://你的域名/api/chat' \
	-H 'Content-Type: application/json' \
	-d '{"question":"新生什么时候选课"}'
```

预期结果：
- 返回 200
- JSON 中包含 `answer` 字段

补充测试：

```bash
curl 'https://你的域名/api/kb_query?q=新生如何选课'
```

预期结果：
- 返回 200
- 包含 `retrieved`、`top_score`、`answer` 字段

### 4.2 页面级测试

1. 打开 `https://你的域名/chat`
2. 发送问题：`新生什么时候选课`
3. 连续发送 3 个不同问题

预期结果：
- 消息可正常收发
- 页面不重复、不丢消息
- 异常网络时有友好提示

### 4.3 FAQ 命中与兜底

1. 提问 FAQ 中已有问题（如校园网）
预期：可命中并返回相关内容

2. 提问 FAQ 外问题
预期：
- 若 DeepSeek 可用，返回模型回答
- 若 DeepSeek 不可用，返回兜底提示（建议查看官网/联系辅导员）

## 7. 问题判定与处理建议

### 7.1 地图问题判定

- 状态码 121：配额超限，等待次日或更换 Key
- 状态码 311：Key 或服务配置问题，检查控制台

### 7.2 DeepSeek 问题判定

- 若日志出现 `LLM request failed`：检查 Key、余额、模型名
- 若日志出现 `LLM request exception`：检查网络与超时
- 若 `/api/demo_status` 显示 `llm_enabled=false`：检查 `.env` 中 `DEEPSEEK_API_KEY`
- 若 `/api/demo_status` 显示 `kb_index_ready=false`：先调用 `/api/rebuild_index`，再检查 `feature2-ai/data` 目录是否有 `.md` 文档
- 若 `.env` 中 `KB_FORCE_LEXICAL=0` 但 `kb_backend` 仍为 `lexical`：检查 HuggingFace 网络连通和后端日志中的 `faiss_build_failed` 错误信息
- 可临时将 `OPENAI_TIMEOUT` 增大后复测
